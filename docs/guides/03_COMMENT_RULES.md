# 주석 규칙 (이 프로젝트용)

## 목표
- “전략은 계속 바뀐다” 전제에서, **왜(의도/가설)** 와 **검증 포인트**를 남긴다.
- 코드 설명(what)은 최소화하고, **근거/리스크/대안(why)** 위주로 작성한다.

---

## 1) 필수 주석(반드시)
- 전략 진입/청산 조건의 **근거**와 **실험 가설**
- 리스크 제약(최대 손실, 최대 주문 횟수 등)의 **선택 이유**
- 체결/우선순위/충돌 처리에서 “왜 이 규칙이 고정인지”
  - 이 부분은 엔진 계약(`../architecture/06_ARCHITECTURE.md`)을 인용해야 한다.

---

## 2) 금지 주석(하지 말 것)
- 코드 한 줄을 그대로 한국어로 바꾸는 주석
- “나중에 고치자” 류의 모호한 메모 (대신 TODO 규칙 사용)

---

## 3) TODO 규칙
- 형식: `// TODO(#issueId): 무엇을, 왜, 완료 조건`
- 완료 조건이 없는 TODO 금지

예:
- `// TODO(#42): 슬리피지 모델을 체결기반으로 교체. 완료조건: 30일 리포트에서 MDD 개선 확인`

---

## 4) 전략 파일 상단 주석 템플릿(권장)
- 전략 개요(한 문장)
- 핵심 가설(1~3개)
- 실패 조건(언제 성능이 무너지는가)
- 파라미터 주의사항(민감도 높은 값)
- 고정 계약 링크:
  - 체결 모델(fillModelRequested/fillModelApplied): `../architecture/06_ARCHITECTURE.md`
  - 파라미터 정의: `../specs/09_PARAMETER_REGISTRY.md`
  - 실험 규격: `../specs/10_EXPERIMENT_PROTOCOL.md`

---

## 5) 변경 이력은 주석이 아니라 “runId 메타”
- 변경 이력/실험 결과는 주석이 아니라 runId 리포트 메타로 남긴다.
- 규격: `../specs/10_EXPERIMENT_PROTOCOL.md`
